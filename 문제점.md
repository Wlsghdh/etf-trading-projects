### 상태
- scraper.py `_get_timeframe_code()` 및 db_service.py `get_table_name()` 매핑 수정 완료
- 수정일: 2026-03-31

### 수정된 매핑
```
12달(1Y) → 1일 간격  → _D (일봉)     ← 기존 유지
1달(1M)  → 30분 간격 → _30m (30분봉)  ← 수정됨 (기존: _D 덮어쓰기)
1주(5D)  → 5분 간격  → _5m (5분봉)    ← 수정됨 (기존: _D 덮어쓰기)
1일(1D)  → 1분 간격  → _1m (1분봉)    ← 수정됨 (기존: _1h)
```

### 참고
- 기존 _D 테이블 데이터에 5분봉/30분봉이 혼재되어 있을 수 있음
- 다음 스크래핑 실행 시 올바른 테이블에 저장됨
- 미수집 timeframe (10m, W, M, 12M)은 예전에 수집 후 방치 상태 (별도 이슈)

## 6. etf2_db_processed 최신화 지연 — 피처 처리 실패

### 현상
- etf2_db_processed의 최신 데이터: **2026-02-25** (약 1달 전)
- etf2_db(원본)는 3/22까지 수집됐지만, processed는 업데이트 안 됨

### 원인 (2가지)
1. **run-pipeline.sh Step 1에서 중단**: 스크래핑 status가 "error"로 남아있어서 파이프라인이 Step 1에서 멈추고 Step 3(피처 처리)까지 안 감 → **수정 완료** (error여도 데이터가 있으면 성공으로 간주)
2. **피처 처리 자체 실패**:
   - `include_macro: true` → **FRED API key 없음** (거시경제 데이터 못 가져옴)
   - `include_macro: false` → **"No data fetched from provider"** (yfinance 등 외부 데이터 소스 실패)

### 해결 필요
- FRED API key 발급 후 scraper-service 환경변수에 추가 (`FRED_API_KEY`)
- 또는 피처 처리 코드에서 외부 데이터 없이 DB 데이터만으로 처리하도록 수정
- 이게 해결되어야 ML 예측이 최신 데이터로 돌아감
