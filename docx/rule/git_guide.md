# Git 가이드 - Git이 뭐고 왜 쓰는가

## 1. Git이 뭐야?

Git은 **코드의 변경 이력을 기록하는 도구**야.

워드에서 "다른 이름으로 저장"을 여러 번 해본 적 있지?

```
보고서_최종.docx
보고서_최종_수정.docx
보고서_진짜최종.docx
보고서_진짜진짜최종.docx
```

Git은 이걸 자동으로 관리해줘. 파일을 하나만 유지하면서도 **모든 변경 이력**을 기억하고, 언제든 과거로 돌아갈 수 있어.

---

## 2. 왜 "브랜치(branch)"라고 해?

branch = **나뭇가지**라는 뜻이야.

나무를 생각해봐:

```
        🌿 feat/scenario     (임대윤이 시나리오 작업 중)
       /
      /   🌿 feat/regression  (최인훈이 회귀 모델 작업 중)
     /   /
🌳 main ── develop ──────────  (팀 통합본)
               \
                \  🌿 fix/scraper  (박성문이 버그 수정 중)
                 \
                  🌿 feat/community (양진우가 커뮤니티 작업 중)
```

- **줄기(main/develop)** = 모든 코드가 모이는 본체
- **가지(branch)** = 각자 독립적으로 뻗어나가는 작업 공간
- **각 가지는 서로 영향을 주지 않아** → 내 가지에서 실수해도 다른 가지는 안전

나무 가지처럼 **줄기에서 갈라져 나온다**고 해서 "branch"라고 불러.

## 3. 왜 브랜치 관리를 해야 해?

### 브랜치 없이 5명이 작업하면?

```
❌ 모두 같은 코드에서 직접 작업

A가 파일 수정 → push
B가 같은 파일 수정 → push → 충돌!
C가 실험적 코드 추가 → 전체 서비스 다운
D가 되돌리려고 함 → A, B 작업까지 같이 사라짐
```

**결과**: 매일 충돌, 서로 코드 덮어쓰기, 롤백 불가능

### 브랜치로 작업하면?

```
✅ 각자 독립된 가지에서 작업

A는 feat/a 에서 작업 → 완성되면 PR
B는 feat/b 에서 작업 → 완성되면 PR
C는 feat/c 에서 실험 → 실패하면 가지만 삭제
D는 fix/d 에서 버그 수정 → 바로 PR

서로의 작업에 영향 ZERO
```

**결과**: 충돌 없음, 안전한 실험, 깔끔한 코드 관리

## 4. 주요 Git 용어 쉽게 설명

### 기본 용어

| 용어 | 쉬운 설명 | 비유 |
|------|-----------|------|
| **Repository (저장소)** | 프로젝트 전체가 담긴 폴더 | 팀 공유 드라이브 |
| **Commit (커밋)** | "이 시점의 코드를 저장" | 게임 세이브 포인트 |
| **Branch (브랜치)** | 독립된 작업 공간 | 나뭇가지 |
| **Checkout** | 다른 브랜치로 이동 | 다른 작업 공간으로 이동 |
| **Push** | 내 작업을 GitHub에 올림 | 클라우드에 업로드 |
| **Pull** | GitHub에서 최신 코드 받기 | 클라우드에서 다운로드 |

### 협업 용어

| 용어 | 쉬운 설명 | 비유 |
|------|-----------|------|
| **Pull Request (PR)** | "내 작업을 메인에 합쳐주세요" 요청 | 결재 올리기 |
| **Merge (머지)** | 두 브랜치를 합치기 | 두 문서를 하나로 통합 |
| **Conflict (충돌)** | 같은 부분을 다르게 수정해서 Git이 혼란 | 같은 칸에 두 사람이 다른 값 적음 |
| **Review (리뷰)** | PR을 검토하는 것 | 결재 전 내용 확인 |
| **Clone** | 프로젝트를 내 컴퓨터에 복사 | 공유 폴더를 내 PC에 복사 |
| **Fork** | 다른 사람의 프로젝트를 내 계정에 복사 | (우리는 안 씀) |

## 5. 브랜치의 생명주기

```
① 생성: develop에서 새 가지를 만든다
   git checkout develop
   git checkout -b feat/my-feature

② 작업: 내 가지에서 코딩한다
   (코드 수정 → commit → push 반복)

③ PR: 완성되면 develop에 합쳐달라고 요청한다
   (GitHub 웹에서 Pull Request 생성)

④ 리뷰: PM이 코드를 확인한다
   (문제 있으면 수정 요청 → 수정 후 다시 push)

⑤ 머지: PM이 승인하면 develop에 합쳐진다
   (Squash and merge 클릭)

⑥ 삭제: 합쳐진 가지는 삭제한다
   (GitHub에서 "Delete branch" 클릭)
```

## 6. main vs develop 왜 2개야?

```
main     = 완성품이 있는 곳 (서버에서 실행 중인 코드)
develop  = 개발 중인 곳 (팀원들 작업이 모이는 곳)
```

| | main | develop |
|--|------|---------|
| 용도 | 실제 서비스 배포 | 개발 통합 |
| 안정성 | 항상 안정 | 가끔 불안정할 수 있음 |
| 누가 관리 | PM만 | PM + 양진우 |
| 직접 push | 절대 금지 | 금지 (PR로만) |

**왜 나누냐?**
- develop에서 버그가 나도 main(실제 서비스)은 안전
- develop에서 충분히 테스트한 후에 main으로 배포

## 7. Git vs GitHub 차이

| | Git | GitHub |
|--|-----|--------|
| 뭐야 | 코드 버전 관리 도구 | Git 프로젝트를 올리는 웹사이트 |
| 어디서 | 내 컴퓨터에서 실행 | 인터넷 (github.com) |
| 비유 | 카메라 (사진 찍기) | 클라우드 앨범 (사진 공유) |

```
내 컴퓨터 (Git)  ←→  GitHub (원격 저장소)
   commit              push / pull
   (로컬 저장)          (클라우드 동기화)
```

## 8. 자주 하는 실수 & 해결

### "어디 브랜치에 있는지 모르겠어"
```bash
git branch          # * 표시가 현재 브랜치
```

### "다른 브랜치에서 작업해버렸어"
```bash
git stash                         # 변경사항 임시 저장
git checkout feat/내브랜치         # 올바른 브랜치로 이동
git stash pop                     # 변경사항 복원
```

### "커밋 메시지를 잘못 썼어"
```bash
git commit --amend -m "올바른 메시지"   # 직전 커밋만 수정 가능
```

### "파일 수정을 되돌리고 싶어"
```bash
git checkout -- 파일명    # 해당 파일만 마지막 커밋 상태로 복원
```

### "뭘 수정했는지 보고 싶어"
```bash
git status              # 수정된 파일 목록
git diff                # 수정된 내용 상세
```

## 요약

> **Git** = 코드 변경 이력을 저장하는 도구다.
> **Branch** = 나뭇가지처럼 갈라져 나온 독립 작업 공간
> **브랜치 관리 이유** = 5명이 충돌 없이 동시에 작업하려고
> **GitHub** = Git 프로젝트를 공유하는 클라우드
> **PR** = "내 작업 완성했으니 합쳐주세요" 요청
