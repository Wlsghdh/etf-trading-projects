## 슬라이드 3: 데이터 수집 현황

**슬라이드 내용:**
- 전체 1,000개 심볼 (S&P 500 + 중대형주 + ETF)
- 4개 타임프레임: 1Y(일봉), 1M(30분봉), 5D(5분봉), 1D(1분봉)
- 최근 수집: **885/1,000 성공** (88.5%)
- 실패 종목: ~103개 (주로 ETF + 일부 개별주)
- 소요 시간: 약 **12시간** (1,000종목 × 4 타임프레임)
- 자동화: Cron으로 매일 06:00 KST (월~금)

**대본:**
> 현재 1000개 종목 데이터를 TradingView에서 Playwright로 자동 수집하고 있습니다. 1회 수집에 약 12시간이 걸리고, 885개 종목이 성공적으로 수집됩니다. 103개 실패 종목은 TradingView UI 버튼 셀렉터 문제였는데, 이번에 6가지 fallback 방식으로 개선했습니다.

## 슬라이드 4: 데이터 품질 이슈 발견

**슬라이드 내용:**
cih0210 선배가 `etf_63_pred` DB 작업 중 발견한 7가지 문제:

| # | 문제 | 심각도 |
|---|------|--------|
| 1 | _D 테이블에 intraday(5분봉) 데이터 섞임 | 치명적 |
| 2 | 다른 종목 가격이 같은 테이블에 혼입 (CSCO에 $861) | 치명적 |
| 3 | 주식 분할(split) 보정 없음 | 높음 |
| 4 | target_3m이 행 기반이라 intraday 섞이면 의미 깨짐 | 높음 |
| 5 | PRIMARY KEY가 time만 → 데이터 무결성 없음 | 치명적 |

- 예: CSCO_D 테이블에 $0.07~$1,170 가격 (실제 CSCO는 $50~$65)
- 예: AAPL_D에 하루 78행 (일봉이면 1행이어야 함)

**대본:**
> cih0210 선배가 63일 뒤 가격 예측 작업 중에 심각한 데이터 오염을 발견해주셨습니다. 예를 들어 CSCO 테이블에 $861짜리 데이터가 들어있는데, 실제 CSCO는 $50~$65입니다. 다른 종목 데이터가 잘못 들어간 겁니다. 또 일봉 테이블에 5분봉 데이터가 섞여서 하루에 78행이 있는 경우도 있었습니다.

## 슬라이드 5: 데이터 품질 - 해결

**슬라이드 내용:**
| 해결 내용 | 규모 |
|-----------|------|
| PK 복합키 마이그레이션 | **3,460개 테이블** `(time)` → `(symbol, timeframe, time)` |
| 가격 이상치 삭제 | **339,220행** (중앙값 대비 5배 이상 차이) |
| intraday 데이터 삭제 | **48,239행** (하루 1행만 유지) |
| 스크래퍼 타임프레임 매핑 수정 | 레거시: "1달"→"D" ❌ → "1달"→"30m" ✅ |
| CSV 검증 로직 추가 | 타임프레임 자동 감지 + 가격 이상치 필터링 |
| 심볼 선택 검증 | 차트 타이틀에서 심볼 확인 후 다운로드 |
| split 보정 | yfinance 분할 이력으로 읽기 시 자동 보정 |

**대본:**
> 이 문제들을 전부 해결했습니다. 가장 큰 작업은 3,460개 테이블의 PRIMARY KEY를 복합키로 변경한 것과, 이상치 데이터 약 38만 행을 삭제한 것입니다. 앞으로 새로 수집되는 데이터는 CSV 검증 로직이 적용되어 이런 문제가 재발하지 않습니다.


**대본:**

## 슬라이드 9: 펀더멘탈 데이터 수집 전략

**슬라이드 내용:**
**왜 필요한가?**
- 현재: 기술적 지표만 (RSI, MACD, 이동평균선 등 95개 피처)
- 추가: 회사 재무 상태 (PER, ROE, 매출성장률, 부채비율 등 18개 피처)

**수집 계획:**
| 단계 | 소스 | 비용 | 시기 |
|------|------|------|------|
| 1단계 | yfinance | 무료 | 즉시 |
| 2단계 | FMP API | $29/월 | 한계 도달 시 |

**수집 주기:**
- 매일: PER, PBR, 시가총액, 배당수익률 (~8분/1000종목)
- 매주: ROE, ROA, 매출성장률, 영업이익률 (~17분)
- 분기별: 재무제표 (매출, 순이익, 부채비율, 현금흐름)

**저장:** MySQL `fundamental_daily` / `fundamental_quarterly` 테이블

**대본:**
> 펀더멘탈 데이터 수집 전략입니다. 현재 ML 모델은 기술적 지표만 사용하는데, 회사의 실제 재무 상태를 반영하면 예측 정확도가 올라갈 것으로 기대합니다. 1단계로 yfinance를 사용하면 무료로 1000종목 데이터를 약 8분에 수집할 수 있습니다. PER, ROE, 매출성장률 등 18개 지표를 수집 주기별로 나눠서 관리할 계획입니다.

## 슬라이드 10: 펀더멘탈 - API 연결 구조

**슬라이드 내용:**
```
yfinance API
    ↓
scraper-service/app/services/fundamental_service.py (신규)
    ↓
MySQL: fundamental_daily / fundamental_quarterly
    ↓
ml-service: 피처 파이프라인에 추가 (95 → 113 피처)
    ↓
멀티AI: Fundamental 프롬프트에 실데이터 주입
```

**구체적 실행 일정:**
| 주차 | 작업 |
|------|------|
| 4/14~4/18 | fundamental_service.py 구현 + DB 테이블 생성 |
| 4/21~4/25 | 1000종목 수집 테스트 + Cron 연동 |
| 4/28~5/02 | ML 피처 추가 + 모델 재학습 |

**대본:**
> 구현 경로입니다. scraper-service에 fundamental_service.py를 새로 만들어서 yfinance에서 데이터를 수집하고 MySQL에 저장합니다. 그 데이터를 ML 피처 파이프라인에 추가하면 기존 95개 피처에 18개가 추가되어 113개 피처로 학습합니다. 멀티AI 페이지에서도 AI 프롬프트에 실제 PER, ROE 값을 넣어서 더 정확한 분석을 제공할 예정입니다.

## 슬라이드 12: 매매전략 페이지

**슬라이드 내용:**
3개 탭:
1. **시나리오 예측**: 종목+날짜 선택 → 5가지 시나리오 차트 (급등/상승/보합/하락/급락)
2. **가상 매매**: 초기 자본 설정 → 매수/매도 → 수익률 추적 (localStorage)
3. **매매 복기**: KIS 실제 체결 내역 + 미실현 손익

**대본:**
> 매매전략 페이지에는 3개 탭이 있습니다. 시나리오 예측에서는 종목과 날짜를 선택하면 Geometric Brownian Motion 기반으로 5가지 시나리오 차트를 생성합니다. 가상 매매 탭에서는 실제 돈 없이 투자 연습을 할 수 있고, 매매 복기 탭에서는 실제 KIS 체결 내역과 미실현 손익을 확인할 수 있습니다.
